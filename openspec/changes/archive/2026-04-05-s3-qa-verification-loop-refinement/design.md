# Design: S3 QA Verification Loop Refinement

## Context

当前 `SKILL.md` 中 S3 问答循环的执行流程为线性顺序：
```
回答 → verify → 回显验证结果 → memory update → 下一轮或返回 S2
```

实际使用中发现的问题：
1. Agent 机械执行流程，先抛出答案再验证，验证失败时未立即修正
2. 验证通过后，额外发起一轮"下一轮动作提问"，导致交互冗余
3. 用户期望收到的是"已验证的最终答案 + 下一步引导"的合并消息

## Goals / Non-Goals

**Goals:**
- 将验证 - 修正循环内嵌到回答生成阶段，确保输出给用户的答案已是验证通过的最终版本
- 合并"最终回答文案"和"下一轮动作提问"为单条消息，减少交互轮次
- 保持现有证据验证脚本和记忆脚本接口不变

**Non-Goals:**
- 不改变证据验证脚本的行为或接口
- 不改变记忆引擎的 schema 或写入逻辑
- 不改变回答模板的结构（问题理解、事实、证据、推断、不确定点、证据列表、验证结果）

## Decisions

### Decision 1: 内嵌验证循环模式

**选择**: 采用"草案→验证循环→定稿"模式

```
1. 组织回答草案（含结构化证据数组）
2. 执行 verify
3. 若 verdict == pass:
     草案提升为最终文案
   若 verdict == fail:
     撤回未验证断言，修正草案，返回步骤 2
   若 verdict == uncertain:
     标记不确定点，草案提升为最终文案（继续但告知用户）
4. 输出最终文案 + 下一轮动作提问（合并为单条消息）
5. memory update
```

**替代方案考虑**:
- 保持线性流程，但在"验证失败"时立即修正并重试 → 不足：仍需额外的下一轮提问
- 将验证移至回答生成前 → 不足：无法先验知道哪些证据能通过验证

** rationale**: 内嵌循环确保用户收到的答案已是验证通过的版本，同时合并输出减少交互轮次。

### Decision 2: 最终输出合并策略

**选择**: 将"最终回答文案"和"下一轮动作提问"合并为单条消息

输出结构：
```markdown
[固定回答模板内容]
- 问题理解
- 事实
- 证据
- 推断
- 不确定点
- 证据列表
- 验证结果

[下一轮动作引导]
- 如有需要澄清的点：提出最小化澄清问题
- 否则：询问用户是否继续追问其他问题或进入下一意图阶段
```

**替代方案考虑**:
- 保持分两条消息发送 → 不足：增加交互轮次，体验割裂
- 仅输出回答，不询问下一步 → 不足：失去意图确认环节，可能偏离用户目标

**Rationale**: 合并后用户一次性获得完整信息和明确的下一步引导，符合自然对话节奏。

### Decision 3: 验证失败时的内部重试次数限制

**选择**: 设置最多 3 次内部重试上限

```
最大重试次数 = 3
若连续 3 次验证仍 fail:
  - 输出当前可验证部分
  - 明确告知用户哪些断言无法验证
  - 标记为"需要用户澄清或补充材料"
```

**Rationale**: 避免无限循环，同时在确实无法验证时给用户透明的反馈。

## Risks / Trade-offs

**Risk 1**: Agent 可能仍会机械执行，忽略内嵌循环
- **Mitigation**: 在 SKILL.md 中使用明确的步骤编号和条件分支描述

**Risk 2**: 合并输出可能导致消息过长
- **Mitigation**: 在"问题与考量"中强调简洁性，下一轮提问保持简短

**Risk 3**: 验证失败重试可能消耗更多 token
- **Mitigation**: 设置 3 次重试上限，超限后降级处理

## Migration Plan

1. 修改 `SKILL.md` 中 S3 阶段的"操作步骤"和"固定顺序执行"部分
2. 更新"离开本阶段前必须确认"的检查项
3. 无需脚本迁移，无需记忆数据迁移
