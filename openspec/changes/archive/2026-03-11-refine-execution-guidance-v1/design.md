## Context

项目当前的 Skill 已有明确的主流程和脚本契约，但阶段说明偏简洁，强调语气也不够强，导致部分 Agent 在长链路执行中容易忽略关键约束，或在该问用户时继续擅自推进。

## Goals / Non-Goals

**Goals:**
- 强化关键约束的可见性和约束力。
- 让 S0-S5 每个阶段都具备可执行的步骤、风险和提问触发条件。
- 让两个脚本的 `instructions` 更像执行指南，而不只是字段说明。

**Non-Goals:**
- 不新增新脚本或新模式。
- 不改变主状态机的阶段定义。
- 不改变 `update/read/verify` 的职责边界。

## Decisions

1. 对关键规则统一使用显式强调写法，例如 `**必须**`、`**不得**`、`**关键**`。
   理由：让高优先级约束在长文档中仍然显眼。

2. 覆盖全部 S0-S5，每个状态统一包含三层说明：`操作步骤`、`问题与考量`、`何时询问用户`。
   理由：降低 Agent 自由发挥空间，减少阶段执行不完整问题。

3. 脚本 `instructions` 只补充说明层，不重构 CLI 或复杂协议体系。
   理由：本次目标是提升执行遵循能力，而不是重新设计脚本接口。

## Risks / Trade-offs

- [Risk] 文档明显变长，阅读成本上升。  
  Mitigation：保持统一结构和显式层级，让信息更容易扫描。
- [Risk] 强调过多会削弱重点。  
  Mitigation：只对真正关键的规则使用强强调，其余保持普通指导语气。
